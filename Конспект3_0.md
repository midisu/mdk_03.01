# Моделирование и анализ программных продуктов

**МДК 03.01** — междисциплинарный курс, формирующий инженерное мышление будущего специалиста. Его цель — научить не просто писать код, а проектировать, оценивать и улучшать программные системы на всех этапах их жизненного цикла.

## Что такое моделирование?

**Моделирование** — это создание упрощённого, абстрактного представления (модели) программной системы до или в процессе её реализации.

Модель отражает:
- **Структуру** — из чего состоит система (классы, компоненты, базы данных);
- **Поведение** — как работает (сценарии, взаимодействия, состояния);
- **Архитектуру** — как организованы связи между частями (контейнеры, серверы, микросервисы).

### Основные цели моделирования:
- Визуализировать архитектуру и логику работы;
- Выявить противоречия и ошибки на ранних этапах;
- Обеспечить общее понимание между заказчиком, аналитиками и разработчиками;
- Снизить стоимость исправлений и риски сбоев.

### Основной инструмент: UML
**UML (Unified Modeling Language)** — графический язык для описания систем:
- **Структурные диаграммы**:  
  - Диаграмма классов  
  - Диаграмма компонентов  
  - Диаграмма развёртывания  
  - ER-диаграммы (ERD)
- **Поведенческие диаграммы**:  
  - Диаграмма вариантов использования (Use Case)  
  - Диаграмма последовательностей  
  - Диаграмма состояний  
  - Диаграмма деятельности

## Что такое анализ?

**Анализ** — это систематическое исследование моделей, требований или уже существующего кода с целью:
- Проверки соответствия требованиям;
- Оценки качества, надёжности, безопасности и производительности;
- Выявления ошибок, уязвимостей и технического долга.

### Основные направления анализа:

1. **Анализ требований**  
   — сбор, систематизация и проверка полноты/непротиворечивости.

2. **Статический анализ**  
   — проверка кода без запуска (SonarQube, ESLint, Pylint, Checkstyle).

3. **Динамический анализ**  
   — исследование поведения программы во время выполнения (JProfiler, VisualVM, Valgrind).

4. **Тестирование**  
   — функциональная и нагрузочная проверка (Selenium, Postman, JMeter).

5. **Анализ безопасности**  
   — поиск уязвимостей (OWASP ZAP, Snyk, Checkmarx).

6. **Архитектурный и производительностный анализ**  
   — оценка масштабируемости, отказоустойчивости, времени отклика.

### Метрики качества:
- Цикломатическая сложность  
- Коэффициент связности  
- Время отклика  
- Покрытие тестами

## Задачи и методы моделирования и анализа программных продуктов

### 1. Задачи моделирования
- Визуализация архитектуры и логики системы
- Выявление противоречий на ранних этапах
- Снижение рисков и затрат на исправления
- Обеспечение общего понимания системы

### 2. Методы анализа
- **Выбор критериев сравнения**
  - Определение ключевых параметров для сопоставления программных продуктов
  - Установление приоритетов среди критериев
  - Формирование системы оценки

- **Сравнительный анализ программных продуктов и средств разработки**
  - Систематическое сопоставление характеристик различных решений
  - Оценка функциональности, производительности, удобства использования
  - Примеры объектов сравнения: браузеры, офисные пакеты, средства просмотра видео

- **Обратное проектирование (Reverse Engineering)**
  - Восстановление структуры и логики программы по готовому исполняемому коду
  - Дизассемблирование — преобразование машинного кода в ассемблерный
  - Применение отладчиков и дизассемблеров (OllyDbg, WinDbg, IdaPro)
  - Защита программ от исследования — методы противодействия обратному проектированию

- **Анализ потоков данных**
  - Исследование передачи данных внутри программы
  - Выявление потенциальных утечек информации
  - Оптимизация использования данных

- **Работа с проектной документацией**
  - Разработка документации с использованием графических языков спецификаций
  - Создание и изучение возможностей репозитория проекта
  - Экспорт настроек в командной среде разработки для обеспечения воспроизводимости

- **Ревьюирование (Review)**
  - **Цели ревьюирования**: улучшение качества кода, передача знаний, выявление дефектов
  - **Задачи ревьюирования**: проверка соответствия стандартам, обнаружение ошибок
  - **Этапы ревьюирования**: планирование, подготовка, проведение, исправление, верификация
  - **Объекты ревьюирования**: исходный код, проектная документация, архитектурные решения
  - **Инструментальные средства ревьюирования**: специализированные системы для проведения код-ревью
  - **Планирование ревьюирования**: определение сроков, участников, объёма работы

- **Системы контроля версий**
  - **Механизмы и контроль внесения изменений в код**
  - **Цели использования**: отслеживание истории изменений, совместная работа, откат
  - **Задачи систем контроля версий**: хранение версий, разрешение конфликтов, ветвление
  - **Этапы работы**: коммит, пул, мерж, разрешение конфликтов
  - **Объекты контроля**: исходный код, документация, конфигурационные файлы

- **Исследование программного кода**
  - Проверка на предмет ошибок и отклонений от алгоритма
  - Проверка целостности программного кода
  - Программные измерительные мониторы — инструменты для сбора метрик в реальном времени

- **Метрики программного продукта**
  - **Метрики сложности**: оценка структурной сложности кода
  - **Метрики стилистики**: соответствие код-стайлу и стандартам оформления
  - **Направления применения метрик**: оценка качества, прогнозирование затрат, планирование сроков
  - **Определение метрик программного кода специализированными средствами**
  - **Выполнение измерений характеристик кода в среде** (например, Eclipse C/C++, VisualStudio)
  - **Применение стандартных метрик по прогнозированию затрат, сроков и качества**
  - **Использование метрик программного продукта** для принятия управленческих решений

- **Оптимизация программного кода**
  - Выполнение оптимизации с использованием специализированных программных средств
  - Повышение производительности и эффективности кода

- **Измерительные методы оценки программ**
  - **Назначение**: количественная оценка характеристик программного обеспечения
  - **Условия применения**: зависимость от типа проекта, стадии разработки, доступных данных

- **Эталоны и методы проверки корректности**
  - Стандартные образцы для сравнения
  - Методы верификации и валидации программ

- **Корректность программ**
  - Соответствие программы заданным требованиям и спецификациям
  - Отсутствие ошибок и нежелательного поведения

- **Методы организации работы в команде разработчиков**
  - Распределение ролей и ответственности
  - Организация процессов взаимодействия
  - Использование методологий разработки

- **Представление результатов сравнения**
  - Визуализация данных сравнения
  - Формирование отчётов и выводов

- **Примеры сравнительного анализа программных продуктов**
  - Конкретные кейсы сравнения аналогичных решений
  - Анализ преимуществ и недостатков

- **Современные стандарты качества программного продукта и процессов его обеспечения**
  - Международные стандарты (ISO/IEC 25010, CMMI)
  - Рекомендации по обеспечению качества
  - Процессы контроля и улучшения качества

## Связь моделирования и анализа

Эти процессы взаимосвязаны и дополняют друг друга:
- Моделирование создаёт «чертёж» системы;
- Анализ проверяет этот чертёж на прочность, логичность и соответствие целям.

Моделирование без анализа — красивые, но бесполезные схемы.  
Анализ без модели — догадки без основы.

Вместе они позволяют:
- Управлять сложностью крупных систем;
- Улучшать коммуникацию в команде;
- Создавать качественную документацию;
- Принимать обоснованные технические решения;
- Снижать затраты на поддержку и переделки.

## Где применяется?

- Разработка корпоративных информационных систем;
- Проектирование микросервисных архитектур;
- Авиация, медицина, финтех — где критична надёжность;
- Реинжиниринг и документирование legacy-систем;
- Подготовка к разработке через Agile, Waterfall или спиральные модели.
